## Learned

**Final text manipulation commands:**

| Command | What it does |
|---------|-------------|
| `sort file.txt` | Sorts lines alphabetically or numerically |
| `sort -r file.txt` | Sorts in reverse order |
| `uniq file.txt` | Removes consecutive duplicate lines |
| `uniq -u file.txt` | Shows only lines that appear exactly once |
| `uniq -c file.txt` | Counts occurrences of each line |
| `tr 'a-z' 'A-Z'` | Translates characters, lowercase to uppercase |
| `wc file.txt` | Counts lines, words and characters |
| `wc -l file.txt` | Counts lines only |
| `nl file.txt` | Adds line numbers to output |
| `grep "word" file.txt` | Filters lines containing a specific word |
| `strings file.txt` | Extracts human readable text from any file including binary |

Key insight on `uniq` - the command alone only removes
consecutive duplicate lines. The file must be sorted first
for it to work properly across the whole file, and `-u`
is needed to show only truly unique lines.

Key insight on `strings` - different from `file` which
identifies file type, `strings` actually extracts and
displays all readable text embedded inside any file
including binary files. Extremely useful for finding
hidden content in files that cat cannot read.

**Bandit Level 7 - solved:**

Password stored next to a specific word in a large file
with thousands of lines. One command found it instantly:

```bash
grep "millionth" data.txt
```

**Bandit Level 8 - solved:**

Password is the only line that appears exactly once in
a file with many duplicate lines:

```bash
sort data.txt | uniq -u
```

`sort` first to group duplicates together, then `uniq -u`
to return only the line with no duplicates.

**Bandit Level 9 - solved:**

Password hidden inside a binary file among non readable
content. `cat` returned garbage, `file` confirmed it was
not plain ASCII. Found the solution by searching for a
command that extracts readable text from binary files:

```bash
strings data.txt | grep "=="
```

`strings` extracted all human readable content, piped
into `grep` to filter lines starting with `==` which
the level description hinted at.

## Struggles
`uniq` without `-u` flag does not return unique lines,
it just removes adjacent duplicates. Easy fix once the
distinction was clear but worth remembering - always
`sort` before `uniq` and use `-u` when looking for
lines that appear only once.
